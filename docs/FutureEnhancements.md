# Future Enhancements

A running list of ideas for future versions of Pocket Scanner, organized by intended release. Items here are *candidates*, not commitments ‚Äî we may drop, defer, or reshape any of them.

Versions earlier than the current shipping release are deleted from this doc as they ship; the history is in git.

---

## Next up ‚Äî planned (from 2026-07-07 post-v2.4 review)

Three features, sequenced **C ‚Üí B ‚Üí A**, one per release.

- ~~**C ‚Äî Reduce scanned-PDF file size (v2.5 / 24)**~~ ‚Äî **Built (branch `feature/pdf-file-size`).** `PDFAssembler.renderPage` now downsamples (long-edge cap 2400 px, never upsampling) + JPEG-compresses (quality 0.65) each page, embedding the JPEG as a DCTDecode stream via `PageImageCompressor` + `CGImageSource` ‚Äî **spike confirmed** CoreGraphics embeds it small (a 12MP gradient page ‚Üí PDF < 800 KB, was multiple MB). Because `renderPage` is the sole image‚ÜíPDF path, this also compresses edited + added pages. Searchability preserved (invisible OCR text layer untouched; test asserts `pdf.string`). New pure `PageImageCompressor` (downsample + JPEG). NOT changed: page dimensions/crop, existing docs (new scans only), no user setting. Spec/plan under `docs/superpowers/` dated 2026-07-07. **Tuned on device:** a real T3 slip went 3.5 MB ‚Üí ~745 KB at 2400 px but small text read "soft," so the long-edge cap was raised **2400 ‚Üí 2800** (crisp; ~1 MB/page, still ~3.5√ó smaller). Ready to archive as **v2.5 (24)** (What's New in `marketing/app-preview/v2.5/`). Further size wins come free with B's grayscale default.
- **B ‚Äî Default scan filter + optional stronger "Document" preset (v2.6 / 25).** Finding: scans default to **Color (`.none`)** ‚Üí washed out, and you must change it every scan (racy vs auto-capture). NOTE: applying filters **when editing already exists** (`PageEditorView` has a filter picker + "apply to all pages") ‚Äî that half is done. So the gap = **no persistent default**. Reframe (respecting "scanner, not editor ‚Äî presets, not sliders"): add a **settable default filter** (Settings), optionally **one crisper Document preset** (could reuse `SignatureProcessor`'s flat-field trick) ‚Äî **NOT** Camera-app sliders. `ImageFilter` presets today: none/greyscale/blackAndWhite/photo via `CIColorControls`.
- **A ‚Äî Signature iCloud sync (v2.7 / 26).** Finding: signatures live in **Application Support** (`SignatureStore`) ‚Äî local-only, per-install; delete/reinstall wipes them (why they "vanished"). Fix: move signature storage into the iCloud container with local fallback (reuse the `ICloudContainer` pattern documents use), migrate existing local signatures on first run, and place them **hidden from the scan library / Files** (not under `/Documents`). Store was built for this ("a future iCloud move [is] a localized change"). Rounds out the signatures project.

## Enhancements v1.3 and beyond

Lower priority. Some of these may never ship. The list exists to capture what we considered.

### Documents

- ~~**Copy text from a scan**~~ ‚Äî **Dropped 2026-06-23: already delivered by iOS.** The viewer's invisible OCR text layer is selectable, and `MarkupPDFView.buildMenu` calls `super.buildMenu` (keeping the system edit menu) and only *appends* Highlight/Strikethrough. So the OS already provides **Copy** for a text selection, **Select All ‚Üí Copy** for the whole document (PDFView selects across all pages), plus Look Up / Translate. Both real cases ‚Äî grab specific facts (select snippet) and grab everything (Select All) ‚Äî work today. The only gap is one-tap **"Copy current page"**, which is marginal (you can drag-select a page, and the common cases are covered) and not worth the added menu item. (The earlier "may suppress native Copy" worry was wrong ‚Äî it doesn't.)
- ~~**Merge two documents (v2.2)**~~ ‚Äî **Shipped 2026-06-25.** Long-press a document ‚Üí **Merge into‚Ä¶** ‚Üí pick a target; the source's pages append to the end of the target (lossless ‚Äî OCR text + annotations preserved), the target keeps its name/location, and the source is deleted (confirmed first). Pure `MergeCandidates` (targets = all other non-corrupt docs) + `DocumentMerge` orchestration (append ‚Üí save target in place ‚Üí delete source, in that safe order) + `MergeIntoMenu`; wired into both `LibraryView` and `FolderContentsView` via a shared `MergeAlerts` modifier. Spec: `docs/superpowers/specs/2026-06-25-merge-documents-design.md`.
- ~~**Scan to a chosen folder + one level of nesting (v2.4)**~~ ‚Äî **Built (branch `feature/scan-to-folder`).** The Save sheet (`NameDocumentSheet`) gained a **"Save to"** menu defaulting to the **current context** (root, or the folder/sub-folder you scanned from), letting a scan file into any folder or sub-folder. Added **one level of sub-folders** (root ‚Üí folder ‚Üí sub-folder ‚Üí docs): `FolderContentsView` now shows/navigates sub-folders and can create (**New Sub-folder**) / rename / delete them, **capped at level 2 in the UI** (`FolderPaths.level < 2`; the data layer was already recursive so the cap is UI-only and cheap to relax). Move now targets sub-folders too ("Parent ‚ñ∏ Sub" labels). **Folder creation is library-only** (no inline creation in the Save sheet); **folder search stays shallow** (Main Library search still spans everything). Manual folder setup via the Files app also works (container is document-scope-public, `SupportedFolderLevels = Any`). New pure helpers: `FolderPaths`, `ScanDestinations`; `DocumentStorage.createFolder(named:in:)` / `listFolders(in:)`. Built subagent-driven; spec + plan under `docs/superpowers/` dated 2026-07-04. Motivation: filing tax scans into the strict `Parent ‚Üí T3/T4RSP/Expenses` structure TaxSlipReader consumes.

### Editing

- ~~**Preserve annotations across page edits**~~ ‚Äî **Decided 2026-06-19: warn, don't preserve.** Editing a page (crop / rotate / filter) still rebuilds it via `DocumentMutations.replacePage`, dropping that page's highlights/strikethroughs ‚Äî geometry-remapping was rejected (rare flow; hard/non-affine for crop & perspective; semantically dubious when the marked region is cropped away). Instead the editor now **confirms before discarding marks** (single Apply, plus a note in the Apply-to-all dialog) so the loss is never silent. Shipped in v1.12. (The lossless strip quick-rotate already preserves marks.)
- ~~**Annotation rectangle-drag fallback**~~ ‚Äî **Superseded 2026-06-22 by Signing.** Free-form (non-text-anchored) placement now exists via the signature stamp (`ImageStampAnnotation` placed at arbitrary page coords). If a generic drag-a-rectangle *highlight* is still wanted later, it would reuse the same placement plumbing.

### Signing

- **Sign a document (v2.0)** ‚Äî scan your signature on paper ‚Üí auto-clean to a transparent cut-out (`SignatureProcessor`: B&W ‚Üí key white‚Üíalpha ‚Üí largest-ink-band crop) ‚Üí save a reusable signature (`SignatureStore`, Settings ‚ñ∏ Signature) ‚Üí **Sign** in the viewer drops it as a drag/resize stamp on the page you're viewing; persists as an editable annotation you can **Move** or **Remove**. Built subagent-driven; a front-loaded spike proved image stamp annotations persist *and render* across save‚Üíreload (so no flatten needed).
- **Multiple signatures (v2.x)** ‚Äî `SignatureStore` is now a collection (`<uuid>.png` files, newest-first; legacy single `signature.png` auto-migrated); Settings shows a thumbnail list with swipe-to-delete + Add; **Sign** shows a picker when 2+ exist (places directly for one); the placed annotation carries its signature id in PDF `contents` so **Move** re-places the *same* signature (picker fallback if it was deleted). Local, thumbnail-only (no names). Storage kept iCloud-ready.
- ~~**Single-shot signature capture (v2.1)**~~ ‚Äî **Shipped 2026-06-25** (merged to `main`; ships with the next archive as v2.1). "Add Signature" now opens a single-photo `UIImagePickerController(.camera)` (`Capture/SingleShotCameraScanner.swift`) instead of the multi-page `VNDocumentCameraViewController`, killing the auto-fire multi-capture; document scanning is unchanged. Crop restored via the native **Move & Scale** (`allowsEditing`, fixed-square frame). Three `SignatureProcessor` fixes the raw-photo path exposed (all latent behind VNDocumentCamera's enhancement): honor `imageOrientation` (portrait `.right` capture saved rotated); **flat-field correction** (divide by a blurred illumination estimate) to cancel the lighting vignette that left a halo; and an `inkBounds` ~2% perimeter-margin skip to drop the faint edge rim that inflated the crop. Spec: `docs/superpowers/specs/2026-06-24-single-shot-signature-capture-design.md`.
- ~~**Signature names (v2.3)**~~ ‚Äî **LIVE on App Store as v2.3 (22), 2026-07-03** (bundled with Merge, since v2.2 was rejected/abandoned; see v1-status memory). Each saved signature takes an optional name, edited by tapping its row in Settings ‚Üí **Rename** alert (40-char cap, blank clears), shown under the thumbnail in the "Choose a Signature" picker so multiple signatures are distinguishable at signing time. Names persist in a `names.json` sidecar (`[id: name]`) beside the PNGs ‚Äî PNGs and their newest-first order untouched, no migration, `remove` prunes the entry. Settings row went compact (thumbnail + name/"Add a name" + pencil). Built subagent-driven; spec `docs/superpowers/specs/2026-07-01-signature-names-design.md`, plan `docs/superpowers/plans/2026-07-01-signature-names.md`. **Reordering was dropped** ‚Äî with 2‚Äì3 signatures a custom order is marginal; order stays creation-date, newest first. (Build 22 also carries a fix: the Xcode 26.6 SDK broke the deprecated `MagnificationGesture` used for signature pinch-resize ‚Äî migrated to `MagnifyGesture`.)
- **Other Signing follow-ups (unscheduled):** **iCloud sync** the signatures across devices (storage is sync-ready); typed / on-screen-drawn signatures; initials / date / text stamps; auto-detect the signature line; sign multiple pages at once.

### Error handling

- ~~**Corrupt PDF "Try to recover"**~~ ‚Äî **Dropped 2026-06-19.** All real document saves are atomic + file-coordinated (`DocumentStorage` writes with `data.write(to:options: .atomic)` inside an `NSFileCoordinator`), so an interrupted or failed save never leaves a half-written file ‚Äî the user keeps the complete file or the previous one, never a corrupt one. The only remaining ways an app-created PDF could be unreadable are genuine hardware/filesystem corruption (not reliably recoverable, and bigger problems exist) or a manually sideloaded corrupt PDF (out of scope). The existing üö´ row + confirmed Delete is a sufficient safety net. (Also: PDFKit has no lenient/repair reader, so the original "lenient reader" premise wasn't implementable.) Only adjacent value, if ever: a clear "storage full" save-error message ‚Äî minor.
